브랜치 챕터에서 "커밋이 사라졌어요"의 90%는 reflog로 살린다고 했다. 이번 챕터는 그 안전망을 제대로 깐다. 되돌리기 도구는 여러 개고, **무엇을 어디로 되돌리는지**가 다 다르다. 이걸 구분 못 하면 냥사원처럼 멀쩡한 작업을 날린다.

<Callout tone="success" title="이 챕터의 한 줄 요약">
  되돌리기 도구는 "건드리는 칸"으로 나뉜다. <code>restore</code>는 작업물·스테이징, <code>revert</code>는 과거를 보존한 채 되돌리는 새 커밋, <code>reset</code>은 브랜치 포인터를 옮긴다(<code>--soft/--mixed/--hard</code>로 어느 칸까지 따라갈지 결정). 그리고 무슨 짓을 해도 <code>reflog</code>가 HEAD의 발자취를 다 기억한다.
</Callout>

## 되돌리기 지도

뭘 되돌리고 싶은지부터 정해야 도구가 정해진다.

| 되돌리고 싶은 것 | 명령 | 한 일 |
|---|---|---|
| 작업 디렉토리의 수정 | `git restore <file>` | 스테이징(index)에 있는 버전으로 복원 — add한 적 없으면 결국 마지막 커밋 내용 |
| 잘못 add한 것 | `git restore --staged <file>` | 스테이징만 취소(작업물은 유지) |
| 이미 한 커밋 (히스토리 보존) | `git revert <commit>` | 그 커밋을 무효화하는 **새 커밋** 추가 |
| 커밋 자체를 무르기 | `git reset <commit>` | 브랜치를 과거로 이동 (`--soft/--mixed/--hard`) |
| 날린 커밋 찾기 | `git reflog` | HEAD가 거쳐온 모든 자리 |

핵심 갈림길은 **revert vs reset**이다.

- **revert** — 과거는 그대로 두고, "그걸 취소한다"는 새 커밋을 위에 쌓는다. 히스토리가 남는다. **공유된 브랜치에서 안전.**
- **reset** — 브랜치 포인터를 과거로 옮긴다. 그 사이 커밋들은 갈 곳을 잃는다. 히스토리를 다시 쓴다. **공유 전 내 로컬에서만.**

## reset 3형제 — 어느 칸까지 따라가나

3챕터의 세 칸(작업 디렉토리 / 스테이징 / HEAD)을 떠올리자. reset은 HEAD를 옮기는데, **나머지 두 칸을 같이 끌고 갈지**가 옵션이다.

## 냥사원, 직접 만져보다

과거 커밋 하나를 골라 target으로 잡고 <code>--soft / --mixed / --hard</code>를 번갈아 눌러보자. 같은 reset인데 어느 칸까지 끌려오는지가 눈에 보인다. soft·mixed는 변경을 보존하고, hard만 작업을 떼어낸다.

<ResetPlayground />

<Callout tone="info" title="관찰 포인트">
  <ul>
    <li>과거 커밋을 target으로 고르고 <code>reset --soft</code> → HEAD만 이동. 되돌린 변경이 <strong>staged</strong>로 남는다 (그래서 "여러 커밋을 하나로 합치기"에 쓴다 — 다음 챕터).</li>
    <li><code>--mixed</code>(기본) → 변경이 <strong>작업 디렉토리</strong>로 내려온다.</li>
    <li><code>--hard</code> → 세 칸 전부 이동, 변경 <strong>사라짐</strong>. 냥사원이 누른 게 이거.</li>
    <li>무엇을 하든 아래 <code>reflog</code>에 기록이 남는다 — "여기로 복구"를 누르면 부활한다.</li>
  </ul>
</Callout>

## 작업을 잠깐 치워두기 — stash

되돌리기는 아니지만 같은 식구다. "지금 작업을 커밋하긴 이른데, 잠깐 깨끗한 상태가 필요할 때" 작업물을 서랍에 넣어둔다.

<Callout tone="info" title="stash보다 나은 선택지도 있다">
  핫픽스 때문에 작업을 치워야 한다면 stash 대신 <strong>worktree</strong>(마지막 챕터)가 더 편할 때가 많다. stash는 컨텍스트를 멈춰야 하지만, worktree는 작업을 그대로 둔 채 옆에 새 작업공간을 연다.
</Callout>

## 냥사원이 새로 배운 함정

- **함정 1: `reset --hard`는 작업 디렉토리도 날린다**
  커밋 안 한 변경은 reflog에도 안 남는다 (reflog는 커밋된 것만 추적). `--hard` 전엔 정말 버려도 되는지 두 번 확인.

- **함정 2: 공유된 커밋을 reset하면 안 된다**
  이미 push해서 남들이 가진 커밋을 reset으로 무르면, push가 거절되고 히스토리가 갈라지는 사고가 난다(5챕터의 그 빨간 글씨). 공유된 건 **revert**(새 커밋으로 무효화)로.

- **함정 3: reflog는 영원하지 않다**
  도달 가능한 항목은 기본 90일, reset --hard 등으로 **도달 불가능해진 커밋은 기본 30일** 후 정리(gc)된다. 또 reflog는 **로컬 전용** — clone한 새 저장소엔 없다. 그래도 "방금 날린 것"엔 거의 항상 통한다.

- **팁: 헷갈리면 revert가 가장 안전**
  revert는 아무것도 안 지운다. 과거 위에 "취소 커밋"을 쌓을 뿐. 운영에 잘못 나간 커밋을 되돌릴 때 표준이다.
