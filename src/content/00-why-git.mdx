## git이 뭔데요?

한 문장으로:

> **git은 "지금 이 코드의 상태"를 시간순으로 차곡차곡 저장해두는 시스템**

그래서 언제든:
- 어제 작업한 그 시점으로 돌아갈 수 있고
- 누가 언제 뭘 바꿨는지 알 수 있고
- 여러 사람이 동시에 작업한 걸 한 곳에 합칠 수 있다

## 비유: 게임 세이브포인트

RPG 게임을 떠올려보자.

- 보스 만나기 전에 **세이브**한다 → git의 `commit`
- 보스한테 깨지면 **세이브 시점으로 로드** → git의 `reset`
- 두 가지 다른 진행 라인을 동시에 만들어보고 싶으면 **세이브 슬롯 두 개** → git의 `branch`
- 다른 친구가 자기 슬롯에서 진행한 걸 합쳐 오고 싶으면 → git의 `merge`

코드를 짜다 보면 게임보다 더 자주 망친다. 세이브 없으면 매번 처음부터.

## "그럼 Dropbox 같은 거랑 뭐가 달라요?"

좋은 질문. 둘 다 파일을 백업해주긴 하지만 결정적 차이가 있다.

| | Dropbox / Google Drive | git |
|---|---|---|
| 저장 단위 | 파일 하나하나 | **프로젝트 전체의 한 시점** (스냅샷) |
| 변경 이력 | 최근 며칠 ~ 한 달 | **영구 보존**, 모든 커밋이 다 남음 |
| 누가 바꿨나 | 잘 안 보임 | **모든 커밋에 작성자 기록** |
| 여러 명이 같은 줄 수정 | 한쪽이 덮어쓰거나 충돌 파일 생김 | **conflict로 명시 → 사람이 결정** |
| 갈래 작업 | 안 됨 | **브랜치**로 평행 작업 |

요약: Dropbox는 "내가 작업한 파일들의 백업", git은 "프로젝트가 시간에 따라 어떻게 진화했는지의 기록 + 협업 인프라". 같은 도구가 아니다.

## 왜 굳이 배워야 하는가 — 3가지 이유

### 1. 실수 복구

가장 자주 일어나는 일.

- 코드를 잘못 지웠다 → 어제 커밋으로 복원
- 새 기능이 멀쩡하던 기능을 깼다 → 어디서부터 깨졌는지 history에서 추적
- 5시간 작업이 갑자기 다 사라진 거 같다 → reflog로 복구

세이브 없이 코딩 = 보스 앞에서 세이브 안 누르고 싸우는 거. 그런 사람은 없다.

### 2. 협업의 공용어

세상의 거의 모든 IT 회사가 git을 쓴다. 그래서:

- **GitHub** (git 호스팅 서비스) 가 사실상 개발자 이력서 역할
- 새 회사 입사하면 첫 출근에 `git clone` 부터 친다 (오늘 냥냥이처럼)
- 코드 리뷰, PR, 이슈, 모든 협업 워크플로우가 git 위에서 굴러간다

git을 모르면 동료와 같이 코드를 짜는 게 불가능에 가깝다.

### 3. AI 시대에 더 중요해진다

이게 좀 새로운 이유다.

- AI 코딩 도구(Claude Code, Cursor, Copilot 등)가 코드를 자동으로 고친다 → **AI가 망쳤을 때 되돌리는 안전장치가 git**
- AI 에이전트에게 작업을 시킬 때 격리된 worktree에서 돌리면 안전 (Ch.5에서 다룸)
- 코드 변경 이력이 명확하면 AI가 컨텍스트를 더 잘 이해

요약: 사람이 코드 안 쓰는 시대가 와도 git은 살아남는다. 오히려 더 중요해진다.

<Callout tone="info" title="냥선배 한 마디">
  "AI한테 시키더라도 commit은 사람이 직접 찍어. AI는 종종 멀쩡한 코드를 멀쩡하게 망친다. git이 없으면 그 망친 시점을 찾아갈 길이 없어."
</Callout>

## 어렵나요?

솔직히, **처음 일주일은 헷갈린다**. 명령어 이름도 직관적이지 않고 (왜 `checkout`이 브랜치 이동인지...), 에러 메시지도 무뚝뚝하다.

근데 이 가이드의 출발점은 단순하다:

> **명령어를 외우는 게 아니라, "안에서 무슨 일이 벌어지는지" 모델을 만든다.**

모델이 잡히면 새 명령어를 만나도 "이건 .git의 어디를 건드릴까?" 한 줄로 추론할 수 있다. 그게 이 가이드의 약속이다.

## 이 가이드의 학습 경로

5개 챕터, 약 56분.

1. **스냅샷이라는 발상** — git이 변화를 기록하는 방식 (사진첩 모델)
2. **`.git` 안에는 뭐가 있나** — blob/tree/commit, 객체 3종의 정체
3. **브랜치는 사실 이름표일 뿐** — HEAD와 포인터
4. **히스토리를 합치는 두 가지 방법** — merge vs rebase, conflict
5. **worktree — 평행 작업공간** — 한 레포에서 여러 작업 동시에

각 챕터마다 **직접 만져보는 인터랙티브**가 들어있다. 그냥 읽고 끝이 아니라 손으로 모델을 만든다.
