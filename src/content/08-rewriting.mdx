## 냥사원의 'asdf' 커밋 30개

금요일 오전. 냥사원이 일주일치 작업으로 PR을 올렸다. 리뷰어 냥대리가 PR을 연다.

  <Line who="내레이션">10분 뒤. 냥사원이 깔끔하게 정리하고 push하려는데 또 빨갛다.</Line>

2챕터에서 배운 것 하나: **커밋 해시는 내용 전체(tree+parent+메시지+시간)의 해시**다. 그래서 커밋을 "수정"하면 사실은 **다른 커밋이 새로 생긴다**. 히스토리 다듬기는 이 사실 위에 서 있다. 강력하지만, 모르고 쓰면 냥사원처럼 공유 브랜치를 깨뜨린다.

<Callout tone="success" title="이 챕터의 한 줄 요약">
  <code>amend</code>·<code>rebase -i</code>는 커밋을 다듬는 도구지만, 실제로는 <strong>해시가 다른 새 커밋으로 교체</strong>한다. 그래서 깔끔한 PR을 만드는 데 최고지만 — <strong>이미 push해서 공유한 커밋엔 절대 쓰지 않는다.</strong>
</Callout>

## 마지막 커밋만 손보기 — amend

방금 한 커밋의 메시지를 고치거나, 빠뜨린 파일을 끼워 넣을 때.

    { type: 'comment', text: '빠뜨린 파일을 마지막 커밋에 끼워넣기' },
    { type: 'prompt', text: 'git add forgotten.js' },
    { type: 'prompt', text: 'git commit --amend --no-edit' },
  ]}
/>

`amend`는 "수정"처럼 보이지만 **새 커밋을 만들고 기존 걸 버린다** (2챕터에서 예고했던 그 동작). 그래서 이미 push한 커밋을 amend하면 해시가 달라져 또 거절당한다.

## 여러 커밋을 다듬기 — interactive rebase

## PR을 합칠 때 — squash merge

대부분의 팀은 PR을 main에 합칠 때 GitHub의 **Squash and merge**를 쓴다. PR 안의 지저분한 커밋들을 **합치는 시점에 자동으로 하나로** 뭉쳐준다. 그러면 냥사원처럼 직접 rebase로 정리할 필요도 없다 — 개인 작업 브랜치는 어지러워도 되고, main 히스토리만 깨끗해진다.

## 황금률 — 공유한 커밋은 다시 쓰지 않는다

<Callout tone="danger" title="냥사원의 두 번째 사고">
  rebase·amend는 커밋을 새 해시로 교체한다. 이미 push해서 동료가 그 커밋을 받아간 상태라면, 동료의 히스토리와 내 새 히스토리가 갈라진다 → push 거절, 강제로 밀면 동료 작업 충돌. <strong>규칙: 내 로컬에만 있는(아직 push 안 한) 커밋만 다듬는다.</strong> 정 공유 브랜치를 rebase해야 하면 팀에 공지하고 <code>--force-with-lease</code>로.
</Callout>

## 냥사원이 새로 배운 함정

- **함정 1: amend는 공유 후엔 금지**
  로컬에서 방금 한 커밋엔 자유롭게. push한 뒤엔 새 커밋(또는 새 amend)이 거절을 부른다.

- **함정 2: rebase는 모든 해시를 바꾼다**
  부모가 바뀌면 자식 커밋의 해시도 줄줄이 바뀐다(2챕터). 그래서 공유 브랜치 rebase가 위험한 것.

- **함정 3: squash로 너무 많이 뭉치면 이력이 사라진다**
  "왜 이렇게 고쳤지?"를 추적할 단위가 너무 커진다. 의미 단위(기능 하나 = 커밋 하나)가 적당하다.

- **팁: 깨끗한 커밋은 미래의 디버깅을 살린다**
  `git bisect`(버그가 처음 들어온 커밋을 이진 탐색)나 `git blame`은 커밋이 의미 단위일 때 빛난다. 'asdf' 31개로는 아무것도 못 찾는다.
