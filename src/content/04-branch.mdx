냥사원은 브랜치를 아낀다. 코드가 복사되는 줄 알기 때문이다. 이 오해 때문에 매번 한 브랜치에서 여러 작업을 섞어 하다 사고를 친다. 사실 git에서 브랜치는 **거의 공짜**다. 한 번 만들고 버리는 게 자연스럽다.

<Callout tone="success" title="이 챕터의 한 줄 요약">

## 브랜치 = 책갈피

비유는 단순하다.

- **commit**: 책의 한 페이지 (= 챕터 2의 commit 객체)
- **브랜치**: 그 페이지에 끼워둔 **책갈피** (이름 + 페이지 번호)
- **HEAD**: "지금 보고 있는 책갈피" 라벨

`git branch feature` 는 책갈피 하나 새로 만드는 것. 코드 복사 아님.
`git checkout feature` 는 "이제부터 이 책갈피로 보겠다" 선언. 코드 복사 아님.
`git commit` 은 새 페이지 만들고, **지금 보고 있는 책갈피를 그 새 페이지로 옮긴다**.

이 세 줄이 브랜치 동작의 전부다.

## 냥사원, 직접 만져보다

아래에서 브랜치를 만들고, checkout하고, 커밋해보자. 그래프 옆에 작은 라벨이 **HEAD가 어디 가리키는지**, **각 브랜치가 어느 커밋을 가리키는지**를 매번 보여준다.

권장 시나리오:
1. `git branch feature` 만들기 → 두 브랜치가 같은 커밋을 가리킴
2. `git checkout feature` → HEAD가 feature로 이동
3. `git commit` 두어 번 → feature만 앞으로 나감, main은 그대로
4. `git checkout main` → HEAD가 main으로 돌아옴
5. `git commit` → 이번엔 main만 앞으로 나감 (브랜치가 갈라짐)

  </ul>

## detached HEAD — 냥사원의 "어 커밋 사라졌어요"

위 플레이그라운드에서 commit id를 클릭해서 checkout하면 **detached HEAD** 상태가 된다. 책갈피를 잡지 않고 그냥 페이지 번호만 외워서 그 페이지를 보는 셈.

이 상태에서 새 커밋을 만들면 그 커밋을 가리키는 책갈피가 없다 → 다른 브랜치로 checkout하는 순간 git이 "이 커밋 가비지 컬렉션 대상"으로 분류하고, 운이 나쁘면 영영 잃는다.

## 냥사원이 자주 빠지는 함정

- **함정 1: "브랜치를 많이 만들면 레포가 무거워진다"**
  거짓. 브랜치 100개를 만들어도 디스크엔 40바이트 짜리 파일 100개가 추가될 뿐. 객체는 공유된다.

- **함정 2: "브랜치를 지우면 그 위의 커밋들도 다 날아간다"**
  반은 맞고 반은 틀림. 다른 브랜치가 그 커밋들을 안 가리키고 있으면 **참조 잃은 객체**가 되어 결국 GC된다. 그 커밋들을 살리려면 지우기 전 다른 브랜치/태그로 옮겨두면 된다.

- **함정 3: HEAD를 너무 자주 옮긴다고 작업이 손상되진 않는다.**
  checkout 자체는 비용이 거의 없다. 무서워하지 말고 자유롭게 옮겨가며 코드를 보자. 냥사원, 메모.

- **팁: 브랜치 이름 컨벤션**
  `feat/login`, `fix/typo`, `hot/critical` 같은 slash 컨벤션. git은 디렉토리처럼 그루핑해서 보여준다 (`.git/refs/heads/feat/login`).
