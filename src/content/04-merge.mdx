## 두 가지 합치는 방법

챕터 3에서 main과 feat이 갈라졌다. 이걸 다시 합치는 방법은 정확히 두 가지다.

### 1. `git merge` — 합쳤다는 사실을 기록

새 commit 하나를 만든다. 이 commit은 **부모가 둘** — 두 갈래의 끝을 모두 가리킨다. 기존 커밋은 손대지 않는다.

장점: 히스토리가 사실 그대로 남는다.
단점: 그래프가 어수선해질 수 있다 (사람들이 평행하게 작업한 흔적이 다 보임).

### 2. `git rebase` — 갈래가 없었던 것처럼 다시 적용

feat의 커밋들을 main 끝에 **새로 만들어 붙인다**. 결과적으로 히스토리가 일직선이 된다. **단, 기존 feat 커밋의 해시가 바뀐다** (= 다른 객체가 됨).

장점: 깔끔한 일자형 히스토리.
단점: 이미 다른 사람과 공유한 커밋을 rebase하면 **혼란 (다른 사람 입장에선 커밋이 통째로 바뀜)**.

## conflict는 언제 나나

merge든 rebase든 git은 자동으로 합치려고 한다. **3-way merge**라는 알고리즘으로:

- **base**: 두 브랜치의 공통 조상 commit
- **ours**: 현재 브랜치의 변경
- **theirs**: 합쳐오는 브랜치의 변경

같은 줄을 **양쪽이 base와 다르게 수정**했으면 git이 "어느 쪽이 맞는지 모르겠다" 하고 conflict를 표시한다. 그 외엔 대부분 자동.

## 사고 재구성

처음 일화로 돌아가서, 도훈이 무슨 짓을 한 건지 한 줄씩 풀어보자.

1. `git pull` = `git fetch` + `git merge` → 원격 변경을 가져와 머지 시도
2. 동일 파일의 동일 줄을 양쪽이 수정 → conflict
3. 도훈이 conflict marker가 **들어있는 채로** `git add` → marker도 그냥 코드의 일부로 staged
4. `git commit` → marker 포함된 파일이 그대로 commit
5. `git push` → 서버에 marker 들어간 코드가 올라감
6. 빌드/실행 시 `<<<<<<<` 줄에서 syntax error → 서비스 다운 → 사장님 슬랙

<Callout tone="danger" title="실전 점검 — marker를 못 봤다는 게 가장 흔한 사고">
  git은 친절하게 <code>git status</code>에 "Unmerged paths" 라고 표시해주고, 파일 안엔 <code>&lt;&lt;&lt;&lt;&lt;&lt;&lt;</code> 가 박혀있다. 그래도 IDE 한구석에서 못 보면 그대로 commit된다. 안전장치:
  <ul>
    <li>merge/rebase 직후엔 <code>git diff --check</code> 또는 <code>grep -rn '&lt;&lt;&lt;&lt;&lt;&lt;&lt;'</code> 한 번 돌리자.</li>
    <li>pre-commit hook으로 conflict marker가 들어가면 commit 거부하게 막아두자. 도훈은 이걸 그날 저녁 바로 설정했다.</li>
  </ul>
</Callout>

## 도훈이 이번 주에 새로 배운 함정

- **함정 1: `git pull --rebase` 와 그냥 `git pull` 의 차이**
  default는 merge. `--rebase`면 내 로컬 커밋이 원격 끝에 재적용된다. 팀이 깔끔한 히스토리를 원하면 `pull.rebase = true` 설정 추천.

- **함정 2: rebase 도중 conflict가 나면 커밋마다 풀어야 한다**
  merge는 한 번에 모든 충돌이 한 머지 commit으로 모임. rebase는 **각 커밋을 차례로 재적용**하므로 같은 줄이 여러 커밋에서 건드려졌으면 같은 곳을 여러 번 풀어야 할 수 있다. <code>git rebase --continue</code> / <code>--abort</code>.

- **함정 3: "fast-forward"는 merge 같지만 merge commit이 안 만들어진다**
  feat이 main에서 갈라진 뒤 main에 아무 새 커밋이 없으면, main을 그냥 feat 끝으로 옮기기만 하면 끝. 이걸 fast-forward라고 부른다. 히스토리에 합치기 흔적이 안 남으므로 PR 머지에선 보통 막거나 옵션으로 제어.

- **팁: VSCode/JetBrains의 merge 도구는 진짜 좋다**
  3-way diff를 시각적으로 보여줘서 marker로 정리하는 것보다 훨씬 안전. 도훈은 그 사건 이후로 이걸 매번 켠다.
