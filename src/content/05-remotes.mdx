여기까지 냥사원은 전부 **자기 컴퓨터 안**에서만 git을 만졌다. 그런데 협업은 "내 저장소"와 "서버 저장소"가 어떻게 대화하는지를 아는 데서 시작한다. 그 모델이 없으면 `--force`가 무슨 짓인지도 모른 채 누른다.

<Callout tone="success" title="이 챕터의 한 줄 요약">
  리모트(origin)는 그냥 "다른 곳에 있는 또 하나의 git 저장소"다. <code>fetch</code>는 서버 상태를 가져와 <code>origin/main</code>이라는 사본만 갱신하고, <code>push</code>는 내 커밋을 서버에 보낸다 — 단, 서버가 내 히스토리의 연장선일 때만(fast-forward). 아니면 거절한다. 그게 빨간 글씨다.
</Callout>

## 저장소가 둘이다

`git clone`을 하는 순간 두 개의 완전한 저장소가 생긴다.

- **origin**: 서버(GitHub 등)에 있는 저장소. 팀의 공용 진실.
- **로컬**: 내 컴퓨터에 통째로 복제된 저장소. **히스토리 전부가 내 디스크에 있다.**

여기서 핵심은 세 번째 등장인물이다.

| 이름 | 정체 |
|---|---|
| `main` | 내 로컬 브랜치 (내가 커밋하면 전진) |
| `origin/main` | **내가 마지막으로 본 서버의 main** — 리모트 추적 브랜치 |
| 서버의 `main` | 실제 서버 상태 (동료들도 여기에 push) |

`origin/main`은 서버와 실시간 연결이 아니다. **내가 마지막으로 `fetch`했을 때 찍어둔 사본**이다. 이 셋이 어떻게 어긋나고 다시 맞춰지는지가 협업의 전부다.

`fetch`와 `pull`의 차이가 헷갈리는 신입이 많다. **fetch는 "정보만 받아온다"** (origin/main 갱신, 내 코드 그대로). **pull은 "받아와서 내 작업에 합치기까지"** 한다 (fetch + merge). 안전하게 먼저 보고 싶으면 fetch, 바로 합치겠다면 pull.

## 냥사원, 직접 만져보다

아래에서 로컬과 서버(origin)가 어떻게 어긋나는지 만들어보자. 추천 순서: **동료가 push → 로컬 commit → push(거절됨) → pull(통합) → push(성공)**. 그리고 마지막에 `push --force`가 동료 커밋을 어떻게 지우는지 직접 눌러보자.

<RemoteSync />

<Callout tone="info" title="관찰 포인트">
  <ul>
    <li><code>git fetch</code>를 누르면 <code>origin/main</code> 라벨만 움직인다. 내 <code>main</code>은 그대로. "서버에 새 게 있구나"를 그제야 안다.</li>
    <li>로컬에 내 커밋이 있고 서버에도 동료 커밋이 있으면 → <strong>갈라짐</strong>. 이 상태에서 push하면 거절된다.</li>
    <li><code>pull</code>이 둘을 합쳐 merge 커밋을 만든다. 그 후엔 push가 통과한다.</li>
    <li><code>push --force</code>는 거절을 무시하고 서버를 내 것으로 덮어쓴다 → 동료 커밋이 사라진다. 빨간 경고가 뜬다.</li>
  </ul>
</Callout>

## 사고 재구성 — 냥사원이 누른 것

1. 동료가 먼저 서버 `main`에 커밋을 push했다.
2. 냥사원도 로컬에서 커밋하고 `git push` → 서버엔 냥사원에게 없는 커밋이 있으니 **non-fast-forward**, 거절.
3. 빨간 글씨 = "그냥 덮어쓰면 남의 커밋 사라져요. 먼저 합치세요" 라는 git의 안전장치.
4. 냥사원이 `--force` → "안전장치 무시하고 서버를 내 로컬로 덮어써" → **동료 커밋 소멸**.

<Callout tone="danger" title="--force 는 거의 항상 답이 아니다">
  거절(<code>! [rejected]</code>)의 올바른 대응은 <code>git pull</code>(또는 <code>git pull --rebase</code>)로 서버 변경을 먼저 가져와 합치는 것. 꼭 force가 필요한 드문 경우(내가 rebase로 정리한 <strong>내 개인 브랜치</strong>)엔 <code>--force-with-lease</code>를 써라 — "내가 마지막으로 본 이후 서버가 안 바뀐 경우에만" 강제한다. 남의 작업을 뭉갤 위험이 줄어든다.
</Callout>

## 냥사원이 새로 배운 함정

- **함정 1: "fetch했는데 코드가 그대로예요"**
  정상이다. fetch는 `origin/main`만 갱신한다. 내 작업 디렉토리에 반영하려면 `git merge origin/main` 또는 처음부터 `git pull`.

- **함정 2: `git push` 했는데 "upstream이 없다"**
  로컬 브랜치가 어느 리모트 브랜치를 따라갈지 안 정해진 것. `git push -u origin <브랜치>`로 한 번 연결하면 다음부턴 그냥 `git push`.

- **함정 3: "origin은 진짜 main이다"라는 착각**
  `origin/main`은 사본이라 항상 한 발 늦을 수 있다. 동료가 방금 올렸어도 내가 fetch 전이면 안 보인다. 협업 중엔 작업 전 `git fetch` 습관.

- **팁: 거절당하면 당황 말고 pull**
  `! [rejected]`는 사고가 아니라 사고 직전에 멈춰준 것이다. `git pull` → (충돌 나면 다음 챕터에서) → `git push`. 이 리듬이 협업의 기본 박자.

## 다음 챕터로

> 냥부장: "응. 이번엔 그거 제대로 풀자."
